# OpenClaw 文档与记忆体系说明

> 说明 OpenClaw 的核心文档结构，以及哪些内容适合放在 `md` 文档里，哪些内容更适合进入 `memory`。

---

## 1. OpenClaw 文档体系结构图

从当前项目说明来看，OpenClaw 的核心不是单一某篇 Markdown，而是由“主配置 + 工作空间 + 文档指令”三层组成。

```text
~/.openclaw/
├── openclaw.json
│   └── 主配置
│      - 模型与 provider
│      - gateway
│      - channels
│      - MCP / 安全 / 日志
│
├── workspace/
│   ├── skills/
│   │   └── 能力扩展
│   │      - 每个 skill 的核心说明通常是 SKILL.md
│   │
│   ├── memory/
│   │   └── 长期记忆
│   │      - 用户偏好
│   │      - 项目背景
│   │      - 历史决策
│   │      - 可被语义检索的经验
│   │
│   └── config/
│      └── 工作空间级配置
│
└── logs/
    └── 运行日志
```

### 三层职责

#### 1. 主配置层

核心文件：`~/.openclaw/openclaw.json`

作用：
- 决定 OpenClaw 怎么运行
- 决定接哪个模型、哪个渠道、哪些服务开启
- 属于系统级配置入口

这一层是“运行核心”，不是知识库文档。

#### 2. 工作空间层

核心目录：
- `~/.openclaw/workspace/skills/`
- `~/.openclaw/workspace/memory/`
- `~/.openclaw/workspace/config/`

作用：
- `skills/` 管能力
- `memory/` 管长期记忆
- `config/` 管当前工作空间的附加配置

#### 3. 文档指令层

典型文件：
- `SKILL.md`
- 灵魂模板类 `md`
- 教程、规范、操作说明类 `md`

作用：
- 告诉 OpenClaw“如何做事”
- 规范输出风格、工作方式、调用流程
- 让不同场景有不同的角色和行为约束

### 可以这样理解

```text
openclaw.json
= 系统总开关

skills / SKILL.md
= 会什么、怎么调用能力

memory
= 记住什么

灵魂模板 / 规范文档 / 教程文档
= 用什么方式工作、怎么表达
```

## 2. 哪些内容适合放 md，哪些适合放 memory

这是最容易混淆的一块。一个简单判断标准是：

- `md`：偏规则、模板、说明、知识库
- `memory`：偏长期记忆、偏好、上下文沉淀、历史经验

### 适合放在 md 文档里的内容

这类内容通常是“稳定的、可复用的、希望被人直接阅读和维护的”。

适合放 `md` 的内容包括：

- 安装教程
- 配置说明
- 操作手册
- 使用规范
- 提示词模板
- 灵魂模板
- Skill 使用说明
- 项目约定
- 输出格式模板
- FAQ
- 经验总结文档

这类内容的特点：
- 需要人工维护
- 需要版本管理
- 需要目录结构和命名规范
- 更适合被“直接阅读”

### 适合放在 memory 里的内容

这类内容通常是“运行过程里逐渐沉淀出来的上下文”，重点不在人工排版，而在后续检索和复用。

适合放 `memory` 的内容包括：

- 用户长期偏好
- 某个项目的背景信息
- 业务上下文
- 历史决策依据
- 已验证过的方案
- 某类问题的处理经验
- 反复出现的注意事项
- 用户不想每次重复讲的前置约束

这类内容的特点：
- 更偏“记忆”而不是“文档”
- 更强调长期保存与召回
- 更适合被 Embedding 和语义搜索使用
- 不一定需要强格式化排版

### 不要混放的典型例子

#### 应该放 md，不适合放 memory

- “OpenClaw 安装说明”
- “飞书接入教程”
- “PPT 创意灵魂模板”
- “Chrome 146 浏览器适配说明”

原因：
- 这是标准知识和操作规则
- 需要长期维护版本
- 需要让人直接打开阅读

#### 应该放 memory，不适合只写在 md

- “这个用户做 PPT 时偏好偏商务、少口号”
- “当前项目优先使用飞书，不走 Telegram”
- “这个团队默认输出要简洁，不要大段空话”
- “上次验证过某个模型在这个项目里不稳定，尽量别用”

原因：
- 这是上下文和偏好
- 更适合后续自动召回
- 不应该每次靠人工翻文档找

---

## 实用判断法

如果你拿不准放哪里，可以直接问自己两个问题：

### 问题 1：这份内容是不是要给人反复阅读？

如果是，优先放 `md`。

例如：
- 教程
- 模板
- 规范
- 说明书

### 问题 2：这份内容是不是要让 OpenClaw 以后自动记住？

如果是，优先放 `memory`。

例如：
- 用户偏好
- 项目长期背景
- 历史决策
- 反复复用的上下文

---

## 最佳实践

建议按这个原则管理：

- 稳定知识放 `md`
- 长期偏好放 `memory`
- 运行参数放 `openclaw.json`
- 扩展能力放 `skills/`
- 工作方式和角色约束放模板类 `md`

这样做的好处是：

- 文档结构清晰
- 记忆边界明确
- 后续维护成本低
- 不容易把“教程”“配置”“记忆”混成一团

---

## 一句话总结

`md` 是给人看、给人维护的知识与规则；`memory` 是给 OpenClaw 长期记住、后续靠语义检索自动召回的上下文沉淀。
